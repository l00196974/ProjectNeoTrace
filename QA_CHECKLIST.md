# QA 代码质量审查检查清单

## 1. 代码规范和可维护性

### 1.1 代码风格（PEP 8）
- [ ] 缩进使用 4 个空格
- [ ] 行长度不超过 120 字符
- [ ] 函数和类命名符合规范（snake_case / PascalCase）
- [ ] 常量使用大写字母和下划线
- [ ] 导入语句顺序正确（标准库 → 第三方库 → 本地模块）
- [ ] 避免使用 `import *`

### 1.2 文档字符串
- [ ] 所有公共函数/类都有文档字符串
- [ ] 文档字符串包含参数说明、返回值、异常说明
- [ ] 使用 Google 或 NumPy 风格的文档字符串
- [ ] 复杂算法有详细注释

### 1.3 代码复杂度
- [ ] 函数长度不超过 50 行
- [ ] 圈复杂度不超过 10
- [ ] 避免深层嵌套（不超过 3 层）
- [ ] 避免重复代码（DRY 原则）

### 1.4 类型注解
- [ ] 函数参数和返回值有类型注解
- [ ] 使用 `typing` 模块的类型提示
- [ ] 通过 `mypy` 类型检查

## 2. 性能优化

### 2.1 CPU 推理优化
- [ ] Student Model 推理时间 < 1ms（CPU）
- [ ] 使用模型量化（INT8/FP16）
- [ ] 批处理优化（batch size 调优）
- [ ] 避免不必要的张量拷贝
- [ ] 使用 `torch.no_grad()` 禁用梯度计算

### 2.2 内存使用优化
- [ ] 内存使用 < 2GB
- [ ] 及时释放大对象（del + gc.collect()）
- [ ] 避免内存泄漏（检查循环引用）
- [ ] 使用生成器处理大数据集
- [ ] 避免全量加载数据到内存

### 2.3 API 性能
- [ ] API P99 延迟 < 10ms
- [ ] 使用连接池（数据库、Redis）
- [ ] 实现缓存策略（LRU Cache）
- [ ] 异步 I/O 优化
- [ ] 避免阻塞操作

### 2.4 数据处理优化
- [ ] 使用向量化操作（NumPy/Pandas）
- [ ] 避免循环处理大数据
- [ ] 使用 Parquet 格式存储数据
- [ ] 数据分区和索引优化

## 3. 错误处理和异常捕获

### 3.1 LLM 调用失败处理

## 9. 性能基准

### 9.1 推理性能
| 指标 | 目标 | 测试方法 |
|------|------|----------|
| Student Model 推理时间 | < 1ms | 单样本推理 100 次取平均 |
| 批处理推理时间 | < 10ms/batch | batch_size=32 推理 100 次 |
| 模型加载时间 | < 1s | 冷启动加载时间 |

### 9.2 API 性能
| 指标 | 目标 | 测试方法 |
|------|------|----------|
| P50 延迟 | < 5ms | 1000 次请求统计 |
| P99 延迟 | < 10ms | 1000 次请求统计 |
| QPS | > 1000 | 并发压力测试 |

### 9.3 内存使用
| 指标 | 目标 | 测试方法 |
|------|------|----------|
| 模型内存 | < 500MB | 加载后内存占用 |
| 运行时内存 | < 2GB | 持续运行 1 小时峰值 |
| 内存泄漏 | 0 | 运行 24 小时内存增长 < 10% |

## 10. 审查流程

1. **代码风格检查**：运行 `pylint`, `black`, `isort`
2. **类型检查**：运行 `mypy`
3. **单元测试**：运行 `pytest` 检查覆盖率
4. **性能测试**：运行性能基准测试
5. **安全扫描**：运行 `bandit`, `pip-audit`
6. **手动审查**：重点审查核心算法和关键逻辑
7. **集成测试**：运行端到端测试
8. **文档审查**：检查文档完整性和准确性
